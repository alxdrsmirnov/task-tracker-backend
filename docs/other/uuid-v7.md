# Почему UUID v7, а не BIGINT или UUID v4

## UUID vs BIGINT

Проект — трекер задач с публичным API: ссылки на задачи, воркспейсы, профили
видны в URL и ответах API.

**BIGINT (autoincrement)** раскрывает последовательность и объём данных
(`/workspaces/3`, `/tasks/147`), упрощает перебор идентификаторов.

Вариант с двумя колонками (внутренний `BIGINT` PK + публичный `UUID`) —
избыточная сложность для монолитного приложения: два индекса, маппинг
между слоями, дополнительные миграции — без ощутимого выигрыша
на масштабах этого проекта.

## UUID v7 vs UUID v4

UUID v4 — полностью случайный. При массовых вставках ключи разбросаны
по B-tree индексу PostgreSQL: больше разбиений страниц, хуже кэш-локальность.

UUID v7 (RFC 9562) содержит метку времени в старших битах,
поэтому новые значения **почти монотонно растут**. Это даёт:

- **Лучшую локальность индекса** — вставки идут в «хвост» B-tree,
  меньше случайных записей по диску.
- **Естественную хронологическую сортировку** — `ORDER BY id` возвращает
  записи в порядке создания, что удобно для курсорной пагинации.
- **Упрощённую отладку** — по значению id видно относительный порядок
  создания объектов.

Единственный компромисс: по UUID v7 можно грубо определить время создания
записи. Для трекера задач это не является угрозой.

## Реализация в Prisma

```prisma
id String @id @default(uuid(7)) @db.Uuid
```

- `uuid(7)` — Prisma генерирует UUID v7 на стороне приложения (поддержка с Prisma 5.18.0).
- `@db.Uuid` — PostgreSQL хранит значение как нативный тип `uuid` (16 байт)
  вместо `text` (36+ байт), что экономит место в индексах и ускоряет сравнения.
